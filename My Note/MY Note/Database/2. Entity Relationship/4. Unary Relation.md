ERD-তে relationship-এর **degree** বলতে relationship-এ কতগুলো Entity অংশগ্রহণ করছে তা বোঝায়।

## 1. Unary Relationship (Degree = 1)

একই Entity নিজের সাথেই relationship তৈরি করে।

```text
Employee
    │
supervises
    │
Employee
```

উদাহরণ:

- একজন Employee অন্য Employee-কে supervise করে।
    
- একজন Person অন্য Person-এর parent হতে পারে।
    

এখানে শুধু **Employee** Entity-ই আছে, তাই এটি **Unary Relationship**।


----

# Why go with Unary Relationship 
## 1. Organization Hierarchy

ধরো কোম্পানিতে:

```text
CEO
├── CTO
│   ├── Developer 1
│   └── Developer 2
└── HR Manager
```

যদি Self-Reference না ব্যবহার করো, তাহলে:

```sql
Employee
---------
EmployeeId
Name
ManagerName
```

সমস্যা:

- Manager change হলে অনেক row update করতে হবে।
    
- Referential Integrity থাকবে না।
    
- Manager সত্যিই Employee table-এ আছে কিনা DB নিশ্চিত করতে পারবে না।
    

তাই:

```sql
Employee
---------
EmployeeId
Name
ManagerId
```

অনেক বেশি scalable।

---

## 2. Category Tree

ধরো E-commerce:

```text
Electronics
 ├── Mobile
 │    ├── Android
 │    └── iPhone
 └── Laptop
```

যদি depth আগে থেকে জানা না থাকে, তাহলে:

```sql
Category
---------
CategoryId
Name
ParentCategoryId
```

সবচেয়ে flexible।

নতুন level যোগ করতে schema change করতে হয় না।

---

## 3. Comment System

```text
Post
 ├── Comment A
 │    ├── Reply A1
 │    │    └── Reply A1.1
 │    └── Reply A2
 └── Comment B
```

তুমি তো আগে থেকে জানো না user কত level reply করবে।

তাই:

```sql
Comment
---------
CommentId
Text
ParentCommentId
```

perfect fit।

---

## 4. Menu System

```text
Products
 ├── Mobile
 ├── Laptop
 └── Camera

Services
 ├── Consulting
 └── Training
```

একই table দিয়েই পুরো menu tree তৈরি করা যায়।

## সবচেয়ে বড় কারণ

### ❌ Fixed-Level Design

```sql
Category
---------
CategoryId
Name
ParentCategoryId
GrandParentCategoryId
GreatGrandParentCategoryId
```

এটা terrible design।

কারণ:

- Level আগে থেকে predict করা যায় না।
    
- Schema বারবার change করতে হবে।
    
- Query জটিল হয়ে যায়।
    

---

### ✅ Recursive Design

```sql
Category
---------
CategoryId
Name
ParentCategoryId
```

যত level দরকার তত level support করবে।

---

## Production Systems কেন ব্যবহার করে?

কারণ:

1. **Schema simple**
    
2. **Unlimited hierarchy depth**
    
3. **Referential integrity**
    
4. **Less data duplication**
    
5. **Natural mapping to real-world structures**
    
6. **Easy maintenance**
    

---

তোমার ASP.NET / SQL Server background থেকে বললে, Self-Referencing FK আসলে Database-এর equivalent of a **tree data structure**।

যেমন programming-এ:

```csharp
class Employee
{
    int Id;
    string Name;

    Employee Manager;

    List<Employee> Subordinates;
}
```

Database-এ এর equivalent:

```sql
Employee
---------
EmployeeId
Name
ManagerId
```

অর্থাৎ Unary Relationship ব্যবহার করার মূল কারণ হলো **real-world tree/hierarchy structure-কে relational database-এ সবচেয়ে natural এবং scalable উপায়ে model করা।**
