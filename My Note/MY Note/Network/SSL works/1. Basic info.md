# "Server sends public key in the SSL certificate" বলতে কী বোঝায়?

এর অর্থ হলো:

> **Server তার Public Key-টি SSL Certificate-এর ভিতরে রেখে Client (Browser)-এর কাছে পাঠায়।**

এটি আলাদা করে শুধু Public Key পাঠায় না, বরং **Certificate**-এর অংশ হিসেবে পাঠায়।

```
Server
   │
   │
SSL Certificate
 ┌─────────────────────────┐
 │ Domain: example.com     │
 │ Public Key              │  ← এখানে Public Key আছে
 │ Issuer (CA)             │
 │ Expiry Date             │
 │ Digital Signature       │
 └─────────────────────────┘
   │
   ▼
Browser
```

---

# Public Key কী?

Public Key হল **Asymmetric Cryptography**-এর একটি Key Pair-এর একটি অংশ।

Key Pair:

```
Private Key  ← Server-এর কাছে গোপন থাকে

Public Key   ← সবাইকে দেওয়া যায়
```

উদাহরণ:

```
Server

Private Key
     ▲
     │
Never leaves the server

Public Key
     │
     ▼
Certificate
```

---

# কেন Public Key পাঠানো হয়?

কারণ Browser-কে Server-এর পরিচয় যাচাই করতে হবে।

Browser-এর দরকার:

- Server-এর Public Key
    
- Domain Name
    
- Certificate-এর Signature
    

এই সবকিছু Certificate-এর ভিতরেই থাকে।

---

# Private Key কি পাঠানো হয়?

**কখনোই না।**

```
Server

Private Key
```

এটি সবসময় Server-এই থাকে।

কেউ এটি দেখতে পায় না।

---

# Certificate-এর ভিতরে কী থাকে?

ধরুন `https://example.com`

Certificate দেখতে অনেকটা এরকম:

```
Certificate

Subject:
example.com

Issuer:
Let's Encrypt

Public Key:
MIIBIjANBg...

Valid From:
2025

Valid To:
2026

Digital Signature:
.....
```

এখানে Browser মূলত Public Key-টাই ব্যবহার করে।

---

# Public Key-এর কাজ কী?

TLS 1.2-এর পুরনো RSA Key Exchange-এ:

Browser একটি Secret তৈরি করে।

```
Random Secret
```

তারপর

```
Encrypt(Random Secret,
        Server Public Key)
```

Server-এর কাছে পাঠায়।

শুধুমাত্র Server-এর Private Key দিয়ে এটি Decrypt করা যায়।

```
Encrypted Secret
       │
       ▼
Server

Decrypt

Private Key
```

---

# TLS 1.3-এ কি Public Key-এর কাজ বদলেছে?

হ্যাঁ।

বর্তমানে (TLS 1.3) Public Key মূলত Server-এর পরিচয় (Authentication) প্রমাণ করতে ব্যবহৃত হয়।

Session Key তৈরির জন্য সাধারণত

**ECDHE (Elliptic Curve Diffie-Hellman Ephemeral)**

ব্যবহার করা হয়।

অর্থাৎ

```
Certificate

↓

Public Key

↓

Server Authentication
```

তারপর

```
ECDHE

↓

Session Key
```

---

# তাহলে Public Key পাঠানোর আসল উদ্দেশ্য কী?

দুইটি প্রধান উদ্দেশ্য:

### ১. Server Authentication

Browser বুঝতে পারে

> "আমি সত্যিই example.com-এর সাথেই কথা বলছি।"

---

### ২. Secure Key Exchange (TLS Version-এর উপর নির্ভর করে)

TLS 1.2-এ

```
Public Key

↓

Encrypt Secret
```

TLS 1.3-এ

```
Public Key

↓

Certificate Verify

↓

ECDHE Key Exchange
```

# পুরো Flow

```
Browser

↓

ClientHello

↓

ServerHello

↓

Certificate
    │
    ├── Domain
    ├── Public Key
    ├── CA Signature
    └── Expiry Date

↓

Browser verifies Certificate

↓

ECDHE Key Exchange

↓

Session Key

↓

Encrypted Communication
```

## একটি সহজ উপমা

ভাবুন, Server-এর কাছে একটি **তালা (Public Key)** এবং একটি **চাবি (Private Key)** আছে।

- **Public Key = তালা** → সবাইকে দেখানো যায়, তাই এটি Certificate-এর মাধ্যমে Browser-এ পাঠানো হয়।
    
- **Private Key = চাবি** → শুধুমাত্র Server-এর কাছে থাকে, কখনো বাইরে যায় না।
    

যে-ই Public Key (তালা) ব্যবহার করে কিছু "লক" করবে, সেটি **শুধু Server-এর Private Key (চাবি)** দিয়েই খোলা যাবে।

> **আধুনিক TLS 1.3-এ** এই উপমাটি মূলত Server-এর পরিচয় যাচাই (Authentication) বোঝাতে বেশি প্রযোজ্য। Session Key তৈরির জন্য সাধারণত ECDHE ব্যবহৃত হয়, RSA Encryption নয়।
