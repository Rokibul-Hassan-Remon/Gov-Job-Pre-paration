# 5 . System Performance

Dashboard profiling indicates that **70% of the transaction latency is caused by CPU waiting for memory read.**

**a)** Upgrading the **CPU** or **RAM**  would be more effective in reducing the latency?
    
**b)** If the memory read speed is doubled, will the total latency be reduced by half? Why or why not?


Yes. In fact, this question is almost certainly designed to test whether the candidate knows **Amdahl's Law**.

## Amdahl's Law

It states that:

> **The overall performance improvement of a system is limited by the portion of the system that cannot be improved.**

The formula is

$$\boxed{\text{Speedup}_{overall}=\frac{1}{(1-P)+\frac{P}{S}}}  $$

where

- (P) = Fraction of execution time that can be improved.
    
- (1-P) = Fraction that cannot be improved.
    
- (S) = Speedup of the improved part.
    

---

## Applying it to your question

Given:

- (P=0.70) (70% of latency is memory waiting)
    
- (S=2) (memory becomes twice as fast)
    

Using Amdahl's Law,

$$\text{Speedup}

\frac{1}{(1-0.70)+\frac{0.70}{2}} $$ 


$$\frac{1}{0.30+0.35}  $$


$$\frac{1}{0.65}  $$

$$\boxed{1.538}  $$

This means the system becomes about **1.54× faster**, **not 2× faster**.

---

### Converting to latency

Suppose the original latency is **100 ms**.

New latency

$$\frac{100}{1.538}  $$

$$\approx65\text{ ms}  $$
Exactly what we calculated manually.

---

## What is the examiner actually testing?

The examiner wants to know whether the candidate understands that:

- ❌ Improving one component **does not** make the entire system improve by the same factor.
    
- ✅ The **non-improved portion** still consumes time.
    
- ✅ The **bottleneck** determines the maximum possible improvement.
    

This is one of the most common concepts in:

- System Design
    
- Computer Architecture
    
- Operating Systems
    
- High Performance Computing (HPC)
    
- Performance Engineering
    

---

## A common interview trick

Many candidates think:

> "Memory is twice as fast, so latency becomes half."

The interviewer expects you to say:

> **"No. Only 70% of the latency is affected. The remaining 30% stays the same. According to Amdahl's Law, the maximum overall improvement is limited by the unaffected portion."**

---

### Interview-style answer

> **a)** Upgrading the **RAM** is more effective because profiling shows that **70% of the latency is caused by memory access**, making memory the primary bottleneck.
> 
> **b)** **No.** According to **Amdahl's Law**, doubling the memory speed only improves the **70%** of latency attributable to memory access. The remaining **30%** is unaffected, so the overall latency decreases from **100 ms to 65 ms** (or equivalently, the system achieves about **1.54× speedup**), **not a 2× speedup or a 50% reduction in latency**.

So yes, if this question appeared in a university exam, BCS ICT exam, or a software engineering interview, the intended concept is very likely **Amdahl's Law**. It is a classic way to test whether a candidate understands **system bottlenecks and performance optimization** rather than simply assuming that making one component twice as fast makes the whole system twice as fast.


**Amdahl's Law** হলো **Computer Architecture** এবং **System Performance Analysis**-এর একটি মৌলিক সূত্র। এটি আমাদের বলে:

> **কোনো সিস্টেমের একটি অংশ যতই দ্রুত করা হোক না কেন, পুরো সিস্টেমের সর্বোচ্চ performance improvement সীমাবদ্ধ থাকবে সেই অংশগুলোর দ্বারা যেগুলো উন্নত করা হয়নি।**

অর্থাৎ,

> **A system is only as fast as its slowest (or unimproved) part.**

# কেন Amdahl's Law দরকার?

ধরুন একটি প্রোগ্রাম চলতে **১০০ সেকেন্ড** লাগে।

এর মধ্যে:

- **70 সেকেন্ড** Memory Access-এ যায়।
    
- **30 সেকেন্ড** CPU Computation-এ যায়।
    

এখন তুমি Memory-কে **২ গুণ দ্রুত** করলে।

অনেকেই ভাববে:

> "Memory 2× faster → পুরো Program 2× faster."

এটি **ভুল**।

কারণ CPU-এর 30 সেকেন্ড তো একই থাকবে।

---

## বাস্তবে কী হবে?

আগে

```
Memory      = 70 sec
CPU         = 30 sec
---------------------
Total       =100 sec
```

Memory 2× faster

```
Memory      =35 sec
CPU         =30 sec
--------------------
Total       =65 sec
```

অর্থাৎ

```
100 → 65 sec
```

**পুরো Program অর্ধেক হয়নি।**

এটাই Amdahl's Law-এর মূল ধারণা।

---

# Amdahl's Law Formula
 
$$\boxed{  
\text{Overall Speedup}

\frac{1}  
{(1-P)+\frac{P}{S}}  
}  $$

যেখানে,

- (P) = যে অংশটিকে improve করা হয়েছে (fraction)
    
- (S) = সেই অংশ কত গুণ দ্রুত হয়েছে
    

---

# Formula-এর অর্থ

ধরো,

```
Program Time

██████████████████████████████ 70%
Memory

█████████████                 30%
CPU
```

Memory-ই শুধু improve হবে।

CPU একই থাকবে।

তাই Formula বলে—

```
New Time = Unchanged Part + Improved Part
```

---

# Example 1

Program-এর

- 80% Memory
    
- 20% CPU
    

Memory 4× faster

তাহলে

[  
$$P=0.8  $$
]

[  
$$S=4  $$
]

$$\text{Speedup}
= 
\frac{1}  
{0.2+\frac{0.8}{4}}$$  
$$
\frac{1}{0.4}  
=2.5$$  
অর্থাৎ

পুরো System

**4× না হয়ে** মাত্র **2.5× faster**

---

# Example 2

$$S=2  $$

$$\text{Speedup}

\frac{1}  
{0.30+\frac{0.70}{2}}$$  

$$ \frac{1}{0.65}

1.538  $$
অর্থাৎ

System

**1.538× faster**

# Maximum Limit

ধরো Memory-কে অসীম দ্রুত (Infinite Speed) করে দিলে।

অর্থাৎ


$$S=\infty  $$
তাহলে
$$\frac{P}{\infty}=0  $$
Formula হবে


$$\text{Speedup}

\frac{1}{1-P}  $$

যদি

```
P=70%
```

তাহলে

 
$$\frac{1}{0.3}
=
3.33 $$ 

অর্থাৎ

Memory **অসীম দ্রুত** হলেও

System কখনো

**3.33× এর বেশি দ্রুত হবে না।**

কারণ CPU-এর 30% কাজ এখনো বাকি।

# বাস্তব জীবনের উদাহরণ

ধরো, তুমি অফিসে যাও।

মোট সময় লাগে **60 মিনিট**।

এর মধ্যে

- 40 মিনিট গাড়ি চালাও।
    
- 20 মিনিট ট্রাফিক সিগন্যালে দাঁড়িয়ে থাকো।
    

এখন তুমি Ferrari কিনলে।

Driving Time

```
40 → 20 min
```

কিন্তু

Traffic Signal

```
20 min
```

একই থাকবে।

তাই

```

```

তুমি **60→30** মিনিটে পৌঁছাতে পারলে না।

কারণ Traffic Signal উন্নত হয়নি।

এটাই Amdahl's Law।

# মনে রাখার সহজ লাইন

> **"Optimize the bottleneck first. Improving a non-bottleneck gives little overall benefit."**

এটাই Amdahl's Law-এর সারকথা।

---

## কোথায় Amdahl's Law ব্যবহার হয়?

Software Engineer বা System Engineer হিসেবে এটি খুব গুরুত্বপূর্ণ। এটি ব্যবহৃত হয়—

- **System Design** – Bottleneck শনাক্ত করতে।
    
- **Computer Architecture** – CPU, Cache, Memory উন্নয়নের প্রভাব বিশ্লেষণে।
    
- **Parallel Computing** – কতগুলো CPU Core যোগ করলে বাস্তবে কত Speedup পাওয়া যাবে তা হিসাব করতে।
    
- **Database Performance Tuning** – Query, Disk I/O, Memory বা CPU—কোন অংশ Optimize করলে সবচেয়ে বেশি লাভ হবে তা নির্ধারণ করতে।
    
- **Cloud & Distributed Systems** – Scaling-এর সীমাবদ্ধতা বুঝতে।
    

একজন ভালো Software Engineer শুধু জানেন না **কী দ্রুত করা যায়**, বরং **কোন অংশ দ্রুত করলে পুরো সিস্টেমে সবচেয়ে বেশি প্রভাব পড়বে**—এই সিদ্ধান্ত নেওয়ার জন্যই Amdahl's Law একটি মৌলিক নীতি।
